# Git Daily Reporter 详细技术设计

## 1. 文档目标

本文件用于描述正式桌面客户端的技术架构、模块边界、数据模型、安全设计、调度机制、凭证处理方案和风险控制策略。

它面向的不是“如何跑一个脚本”，而是“如何把这个产品做成一个长期可维护的桌面应用”。

## 2. 系统总架构

建议采用三层结构：

### 2.1 展示层

技术建议：

- `Tauri + React`

职责：

- 账户、凭证、仓库、模板、调度配置界面
- 连接测试界面
- 运行历史和失败诊断界面
- 日报预览和导出界面

### 2.2 应用服务层

技术建议：

- `Rust`

职责：

- 配置读写
- Git 访问封装
- 凭证解密和注入
- 任务调度编排
- 报告上下文构造
- 调用 AI 总结接口
- 错误归一化

### 2.3 持久化层

技术建议：

- `SQLite` 保存结构化业务数据
- `Stronghold` 保存敏感凭证
- `App Data Directory` 保存缓存仓库、模板、运行产物和日志

## 3. 目录与存储规划

桌面客户端在本地建议维护以下目录：

```text
app_data/
├── db/
│   └── app.sqlite
├── secure/
│   └── stronghold.hold
├── cache/
│   └── repos/
│       ├── repo_001.git
│       └── repo_002.git
├── templates/
│   ├── daily-default.md.j2
│   └── weekly-default.md.j2
├── reports/
│   ├── 2026-04-08/
│   └── 2026-04-09/
└── logs/
    └── app.log
```

说明：

- `db/` 保存业务表
- `secure/` 保存加密密钥库
- `cache/repos/` 保存本地镜像仓库
- `templates/` 保存用户模板
- `reports/` 保存生成报告
- `logs/` 保存运行日志

## 5. Git 访问层设计

## 5.1 访问策略

推荐以应用私有缓存仓库为基础。

流程：

1. 用户添加远程仓库
2. 应用在 `cache/repos/` 下创建 mirror 仓库
3. 后续所有 `fetch/log/diff` 都针对 mirror 仓库
4. 生成报告后只输出结构化结果和最终文件

优点：

- 不污染用户工作目录
- 不受本地未提交内容影响
- 自动化行为可预测

## 5.2 Git 命令范围

第一版需要支持的能力：

- `clone --mirror`
- `fetch --all --prune`
- `log --since --until`
- `log --numstat`
- `rev-parse`
- `show-ref`

## 5.3 Rust 侧实现建议

两条路线：

### 路线 A：系统 Git CLI

优点：

- 行为与用户熟悉的 Git 一致
- 调试简单

缺点：

- 凭证注入和输出控制更麻烦
- 需要严控命令参数和环境变量

### 路线 B：`git2` / `libgit2`

优点：

- 凭证回调、证书检查、主机校验更容易内聚到程序里
- 更适合做应用内认证

缺点：

- 某些场景下和系统 Git 行为可能不完全一致
- 实现复杂度更高

推荐：

- 正式版本优先考虑 `git2/libgit2`
- 某些边界功能不足时，再补系统 Git 兜底

这是基于当前能力边界的工程推断。`git2` 官方 crate 支持 credential callbacks 和 certificate check，可作为实现基础。[git2 crate](https://docs.rs/git2/latest/git2/)

## 6. 凭证系统设计

## 6.1 认证模型

每个仓库绑定一个连接方案：

- `transport = ssh`
- `transport = https_token`

不要让前端只看到“账号”，而不知道真实认证方式。

## 6.2 SSH 设计

应用内需要管理：

- 私钥内容
- 公钥指纹
- passphrase
- known_hosts

需要支持的动作：

- 导入现有私钥
- 生成新密钥
- 测试连接
- 首次主机确认
- 主机指纹变更提示

## 6.3 Token 设计

GitHub PAT 场景建议保存：

- token
- 平台类型
- 用户名提示
- 权限范围摘要
- 最近验证时间

GitHub 官方说明，PAT 在命令行 Git 的 HTTPS 流程中可替代密码使用。[GitHub PAT](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/managing-your-personal-access-tokens)

## 6.4 敏感信息存储

建议：

- 数据库只保存元数据
- 私钥、passphrase、token 都进 Stronghold
- UI 不显示明文
- 复制明文时必须二次确认

## 6.5 解锁模型

建议应用引入“主密码”概念。

工作流：

1. 用户首次启动应用，创建主密码
2. 应用使用主密码解锁 Stronghold
3. 敏感操作前检查当前会话是否已解锁
4. 空闲一段时间后自动重新锁定

## 7. known_hosts 和主机可信设计

应用内单独管理 SSH，就必须单独管理主机可信。

推荐策略：

### 首次连接

- 获取服务端 host key
- 展示指纹给用户确认
- 用户确认后写入应用私有 `known_hosts`

### 后续连接

- 校验 host key 是否匹配
- 不匹配时阻断连接并告警

### 企业模式

- 支持导入预定义 `known_hosts`

## 8. 作者过滤设计

日报默认不是“仓库全部活动”，而是“与当前账户相关的活动”。

推荐分两层：

### 默认模式

- 按账户邮箱精确过滤

### 高级模式

- author name 正则
- author email 列表
- branch 限定
- merge commit 是否忽略
- bot commit 是否忽略

## 9. 调度设计

## 9.1 触发方式候选

### 方案 A：应用常驻 + 内部定时器

优点：

- 实现简单

缺点：

- 应用退出后失效

### 方案 B：托盘 + 内部定时器

优点：

- 体验较好

缺点：

- 跨平台行为不完全一致

### 方案 C：系统调度器

Linux:

- `systemd timer`

Windows:

- `Task Scheduler`

优点：

- 最稳定
- 重启后恢复能力更强

缺点：

- 首次安装和权限处理更复杂

推荐：

- 正式版本使用系统调度器
- 应用内只提供可视化配置与诊断

Microsoft 官方提供 `schtasks` 用于 Task Scheduler 管理；systemd timer 适合 Linux 服务型后台任务。[schtasks](https://learn.microsoft.com/en-us/windows-server/administration/windows-commands/schtasks) [systemd.timer](https://www.freedesktop.org/software/systemd/man/latest/systemd.timer.html)

## 9.2 时间窗口策略

推荐至少支持：

- `natural_day`
- `custom_day_boundary`
- `rolling_hours`
- `since_last_success`

每种策略都应在 UI 中能预览出：

- 本次实际开始时间
- 本次实际结束时间

否则用户很难判断是否统计正确。

## 10. 模板系统设计

## 10.1 模板语法

建议继续保留 Jinja2 风格或兼容语法。

模板变量建议统一成结构化对象：

```json
{
  "report_date": "2026-04-08",
  "repo_name": "demo",
  "branch": "main",
  "summary": {
    "commit_count": 3,
    "changed_file_count": 8,
    "total_additions": 120,
    "total_deletions": 40
  },
  "done_items": ["完成认证页面", "修复同步逻辑"],
  "tomorrow_items": ["补齐异常提示"],
  "files": [
    {"path": "src/app.tsx", "additions": 50, "deletions": 12}
  ]
}
```

## 10.2 模板安全

模板是用户可编辑内容，不能允许它随意执行系统命令。

因此需要：

- 使用受限模板环境
- 限制模板函数暴露范围
- 模板错误要可定位

## 11. AI 总结设计

AI 总结不应直接替代原始 Git 结果，而应作为增强层。

推荐流程：

1. Git 采集层产出结构化数据
2. 模板层可直接使用结构化数据生成基础报告
3. AI 可选读取结构化数据，生成：
   - 今日总结
   - 风险说明
   - 明日建议
4. AI 输出必须作为单独字段保存

这样做的好处：

- 即使 AI 不可用，产品也能工作
- AI 幻觉不会污染原始数据

## 12. 错误模型设计

建议把错误做成结构化类型，而不是只保存字符串。

例如：

- `auth_invalid_token`
- `auth_ssh_passphrase_required`
- `ssh_host_key_mismatch`
- `repo_branch_not_found`
- `repo_remote_unreachable`
- `schedule_install_failed`
- `template_render_error`

UI 应当针对这些错误展示可操作建议。

## 13. 关键坑点的最终推荐

### 13.1 凭证管理

最终推荐：

- 应用内统一管理 SSH 和 Token
- 使用 Stronghold 存储敏感数据
- 不依赖系统 `~/.ssh` 和 Git Credential Manager

### 13.2 仓库访问

最终推荐：

- 基于应用私有 mirror 仓库
- 默认按账户邮箱过滤作者

### 13.3 调度

最终推荐：

- 使用系统调度器
- UI 负责可视化配置和测试

### 13.4 总结生成

最终推荐：

- 基础报告由规则生成
- AI 只做增强，不做唯一事实来源

## 15. 开发时特别注意

- 不要把敏感信息写进日志
- 不要把 UI 表单值直接拼进 shell 命令
- 不要把“运行时间”和“统计窗口”混为一谈
- 不要把仓库全部提交默认当成用户日报
- 不要把 AI 输出当成唯一可靠数据
