# Git Daily Reporter 桌面客户端方案设计

## 1. 项目目标

本项目的目标已经从“命令行日报脚本”升级为“跨平台桌面客户端”：

- 支持 `Linux` 和 `Windows`
- 应用内单独管理 Git 账户
- 应用内单独管理 SSH 密钥
- 兼容 `GitHub Personal Access Token`
- 支持多个仓库、多账户绑定
- 支持自定义日报时间和统计时间窗口
- 支持模板化日报输出
- 后续支持 AI 总结增强

核心定位不是通用 Git 客户端，而是：

> 一个面向个人开发者和小团队的“仓库活动采集 + 自动总结 + 自定义报告”的桌面工作台。

## 2. 为什么要做成客户端

你现在的需求已经明显超出单纯脚本：

- 需要可视化配置多个仓库
- 需要管理多个账户和认证方式
- 需要管理 SSH、Token、主机校验
- 需要让用户自定义总结时间和模板
- 需要查看运行历史和失败原因

脚本能验证思路，但长期可用性和可配置性不够。

因此最终产品形态更适合：

- 前端：简洁桌面界面
- 后端：本地服务能力，负责 Git、凭证、调度、报告生成
- 本地数据层：保存账户、仓库、模板、任务历史

## 3. 推荐技术路线

## 3.1 推荐方案

推荐采用：

- 桌面壳：`Tauri`
- 后端核心：`Rust`
- 前端界面：`React` 或 `Vue`
- 本地数据库：`SQLite`
- 密钥存储：`Tauri Stronghold`

推荐原因：

- `Linux` 和 `Windows` 都容易打包
- Tauri 的安全边界比传统桌面 Web 壳更细
- Rust 适合处理 Git、凭证、调度和文件系统
- SQLite 足够应对本地配置和运行记录
- Stronghold 适合做应用内敏感信息加密存储

这是结合当前需求做出的工程推荐，不是唯一正确答案。

## 3.2 不推荐直接继续放大 Python CLI 的原因

虽然 Python 很适合做原型，但如果继续扩成完整桌面客户端，会逐渐暴露出几个问题：

- 凭证与安全存储边界不够自然
- 跨平台桌面打包体验不如 Tauri 成熟
- 后续如果要严格控制命令执行、权限和沙箱，成本会变高
- 前后端边界不清晰时，工程容易变成“脚本堆叠”

因此合理路线是：

- 保留现有 Python CLI 作为原型验证
- 正式客户端重新按桌面架构落地

## 4. 需求拆解

这个项目实际由 7 个子系统组成：

### 4.1 账户系统

负责管理多个 Git 平台账户：

- GitHub
- GitLab
- Gitea
- 自建 Git 服务

每个账户保存：

- 平台类型
- 显示名
- Git 用户名
- 邮箱
- 默认认证方式

### 4.2 凭证系统

负责管理认证信息。

支持两类凭证：

- `SSH Key`
- `HTTPS Token`

每个仓库可以绑定某个账户下的一份凭证。

### 4.3 仓库系统

负责定义需要被统计的仓库。

每个仓库至少需要：

- 远程仓库 URL
- 默认分支
- 绑定账户
- 绑定凭证
- 作者过滤规则
- 本地缓存目录
- 模板方案

### 4.4 调度系统

负责定义什么时候生成总结。

需要区分两个概念：

- 触发时间：什么时候执行任务
- 统计窗口：统计哪一段时间内的提交

例如：

- 每天 21:30 自动生成
- 统计窗口为“当天 00:00 - 21:30”

或者：

- 每天 09:00 自动生成晨报
- 统计窗口为“昨天 18:00 - 今天 09:00”

### 4.5 Git 采集系统

负责同步远程仓库并提取活动信息。

主要动作：

- 初始化本地镜像仓库
- 定时 `fetch`
- 采集指定时间窗口内的提交
- 聚合文件变更统计
- 可选过滤作者

### 4.6 总结与模板系统

负责把原始 Git 活动转换成可读报告。

包含：

- 规则归纳
- 模板渲染
- 可选 AI 总结

### 4.7 审计与日志系统

负责保存：

- 每次运行的时间
- 成功 / 失败状态
- 失败原因
- 生成文件路径
- 使用的仓库、账户、模板和时间窗口

## 5. 客户端关键页面

我建议桌面应用最少包含以下页面：

### 5.1 首页仪表盘

显示：

- 今日是否已生成日报
- 最近一次运行状态
- 最近失败原因
- 待处理异常仓库数

### 5.2 账户管理页

显示：

- 已配置账户列表
- 平台类型
- 用户名 / 邮箱
- 默认认证方式
- 绑定仓库数

### 5.3 凭证管理页

显示：

- SSH Key 列表
- Token 列表
- 最近使用时间
- 是否有效

注意：

- 私钥和 Token 不能明文展示
- 只允许查看摘要和测试连接

### 5.4 仓库管理页

显示：

- 仓库地址
- 绑定账户
- 绑定凭证
- 默认分支
- 作者过滤策略
- 同步状态

### 5.5 调度与模板页

显示：

- 自动执行时间
- 统计窗口规则
- 输出目录
- 报告模板
- AI 总结开关

### 5.6 运行历史页

显示：

- 每次执行记录
- 成功 / 失败
- 失败明细
- 日报输出路径

## 6. 关键设计原则

### 6.1 账户与仓库解耦

不能把“仓库”和“账户”写死成一对一关系。

推荐模型：

- 一个应用里维护多个账户
- 一个账户下维护多份凭证
- 一个仓库绑定一个账户 + 一份凭证

这是因为你后面很可能会同时使用：

- GitHub 个人仓库
- GitHub 公司仓库
- GitLab 私有仓库

### 6.2 凭证与传输协议解耦

SSH 和 GitHub PAT 不是一回事。

仓库连接方案必须支持：

- SSH URL，例如 `git@github.com:owner/repo.git`
- HTTPS URL，例如 `https://github.com/owner/repo.git`

然后再根据 URL 类型选择认证方式：

- SSH URL -> SSH Key
- HTTPS URL -> Token

### 6.3 调度时间和统计时间解耦

这个项目里最容易做错的地方之一，就是把“每天几点执行”和“统计哪一天的数据”混成一个概念。

必须独立建模：

- `run_at_local_time`
- `window_strategy`

例如：

- 自然日
- 自定义日界线
- 最近 N 小时

### 6.4 工作目录与缓存目录解耦

不要直接用用户的工作仓库做统计主对象。

推荐方案：

- 应用为每个远程仓库维护一个本地缓存仓库
- 所有自动同步和日志分析都基于缓存仓库

这样更安全，也避免污染用户工作区。

## 7. 关键坑点与解决方案

下面是我认为最关键的几个坑。

## 7.1 坑点一：用户以为 Token 和 SSH 是同一种认证

### 方案 A：只支持 SSH

优点：

- 简单
- 不用处理 HTTPS Token

缺点：

- 与 GitHub PAT 需求冲突
- 用户在公司网络或某些环境下可能更依赖 HTTPS

结论：

- 不建议

### 方案 B：支持 SSH 和 HTTPS，但由系统 Git 自己处理凭证

优点：

- 开发难度较低

缺点：

- 和“应用内单独管理账户与凭证”的要求冲突
- 凭证来源不透明

结论：

- 不建议

### 方案 C：应用内显式区分 SSH / Token，两套认证模型并存

优点：

- 语义清楚
- 用户可控
- 可视化配置明确

缺点：

- 需要多写一层凭证调度逻辑

结论：

- 推荐

## 7.2 坑点二：应用内管理 SSH，结果污染系统 `~/.ssh`

### 方案 A：直接复用系统 `~/.ssh`

优点：

- 开发快

缺点：

- 与需求冲突
- 会把应用状态和系统状态混在一起

结论：

- 不建议

### 方案 B：应用内保存私钥文件，但每次调用系统 `ssh`

优点：

- 好实现

缺点：

- 仍然要面对文件权限、known_hosts、临时文件暴露等问题

结论：

- 可作为过渡方案

### 方案 C：应用内加密保存密钥，并在 Git 访问层显式注入凭证

优点：

- 边界清晰
- 更接近产品目标

缺点：

- 实现复杂度更高

结论：

- 推荐作为正式方案

## 7.3 坑点三：团队仓库里统计到别人的提交

### 方案 A：按仓库全部提交统计

优点：

- 最简单

缺点：

- 很多场景下日报会完全失真

结论：

- 不建议作为默认

### 方案 B：按账户邮箱过滤

优点：

- 实现简单
- 大多数个人场景够用

缺点：

- 一个账户可能有多个邮箱

结论：

- 适合作为默认第一版

### 方案 C：支持多维过滤规则

过滤维度包括：

- author name
- author email
- committer email
- branch
- remote

优点：

- 可覆盖复杂团队场景

缺点：

- UI 和规则会复杂

结论：

- 推荐做成高级配置

## 7.4 坑点四：定时执行和统计窗口语义不一致

### 方案 A：固定自然日

优点：

- 简单

缺点：

- 不适合夜间工作或跨时区

结论：

- 可做默认预设，但不能唯一支持

### 方案 B：自定义工作日边界

例如：

- 工作日从 05:00 开始
- 每晚 22:00 生成日报

优点：

- 更贴合真实工作节奏

缺点：

- 要引入时间窗口策略模型

结论：

- 推荐

### 方案 C：支持多个模板化时间策略

例如：

- 自然日
- 最近 24 小时
- 自定义开始结束
- 上次执行以来

优点：

- 灵活度最高

缺点：

- 产品和实现都更复杂

结论：

- 推荐做成长线能力

## 7.5 坑点五：关掉窗口后自动任务失效

### 方案 A：只在应用打开时执行

优点：

- 最简单

缺点：

- 基本不满足“自动化”

结论：

- 不建议

### 方案 B：托盘驻留

优点：

- 用户体验较自然

缺点：

- 在 Linux 和 Windows 上行为差异较大
- 用户可能误以为退出了，其实还在运行

结论：

- 可做

### 方案 C：由系统计划任务执行后台任务，桌面客户端只负责配置

Linux:

- `systemd timer`

Windows:

- `Task Scheduler`

优点：

- 最稳定
- 最适合长期运行

缺点：

- 首次配置稍复杂

结论：

- 推荐

## 7.6 坑点六：首次 SSH 连接时主机可信性没处理

### 方案 A：自动接受所有主机指纹

优点：

- 开发最快

缺点：

- 安全性很差

结论：

- 不建议

### 方案 B：首次连接弹窗确认并保存 `known_hosts`

优点：

- 兼顾安全和可用性

缺点：

- 需要做 UI 交互和本地持久化

结论：

- 推荐

### 方案 C：要求用户手工预置主机指纹

优点：

- 安全可控

缺点：

- 对普通用户不友好

结论：

- 只适合企业高安全模式

## 7.7 坑点七：把“明日计划”完全交给 Git 推断

### 方案 A：完全自动生成

优点：

- 看起来很智能

缺点：

- 容易胡说
- 对真实任务推进不可靠

结论：

- 不建议第一版这么做

### 方案 B：手工填写为主，AI 补充建议

优点：

- 结果可控
- 用户接受度高

缺点：

- 需要保留手工输入流程

结论：

- 推荐

### 方案 C：接入 Issue / Todo 系统，再结合 AI 归纳

优点：

- 数据来源更可靠

缺点：

- 集成成本高

结论：

- 推荐作为 V2 / V3

## 8. 我推荐的产品决策

如果是我来主导这个项目，我会这样定：

1. 客户端使用 `Tauri + Rust + React`
2. 本地用 `SQLite` 管配置、任务和日志
3. 敏感凭证用 `Stronghold` 加密存储
4. 仓库使用应用私有缓存目录，不污染工作区
5. 认证同时支持 `SSH` 和 `HTTPS + PAT`
6. 默认按邮箱过滤作者，高级模式支持自定义规则
7. 自动执行交给系统任务计划，客户端只负责配置和诊断
8. 明日计划采用“手工输入 + AI 建议”，不做纯 Git 自动推断

- 单账户
- 多仓库
- SSH / Token 二选一
- 模板输出
- 手动生成

### V2

目标：

- 多账户
- 仓库绑定账户
- 自动调度
- 运行历史
- 作者过滤

### V3

目标：

- known_hosts 管理
- 高级时间窗口
- 多模板
- AI 总结建议

目标：

- Issue / Todo 联动
- 周报 / 月报
