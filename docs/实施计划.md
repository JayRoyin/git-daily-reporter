# Git Daily Reporter 客户端实施计划

> 当前建议路线是：保留现有 Python CLI 作为原型验证成果，正式产品按跨平台桌面客户端重新实施。

## 阶段 0: 原型保留

目标：

- 保留现有 CLI 作为数据采集和模板思路验证
- 让后续桌面客户端实现时有参考样例

保留内容：

- `prototypes/python-cli/src/git_daily_reporter/`
- `prototypes/python-cli/templates/`
- `prototypes/python-cli/tests/`
- 原型文档

说明：

- 这部分不是最终交付形态
- 但它证明了 Git 采集、模板渲染和测试链路是可行的

## 阶段 1: 桌面客户端工程初始化

目标：

- 初始化 `Tauri + Rust + React`
- 建立桌面端目录结构
- 初始化 SQLite 数据层
- 初始化应用设置目录

交付物：

- 桌面壳工程
- 本地数据库初始化脚本
- 基础路由和页面骨架
- 基础日志设施

## 阶段 2: 数据模型和配置系统

目标：

- 实现 `Account / Credential / Repository / Schedule / Template / RunRecord`
- 打通本地数据库读写
- 实现设置页面的 CRUD 基础能力

重点：

- 账户和仓库解耦
- 仓库和凭证解耦
- 时间窗口单独建模

## 阶段 3: 凭证与安全存储

目标：

- 接入 Stronghold
- 实现主密码解锁机制
- 支持导入 SSH 私钥
- 支持保存 GitHub PAT

重点：

- 密文存储
- 空闲自动锁定
- 凭证测试连接
- 禁止敏感信息进入日志

## 阶段 4: Git 仓库访问层

目标：

- 实现 mirror 仓库初始化
- 实现 `fetch`
- 实现按时间窗口采集提交
- 实现文件增删统计

重点：

- 优先基于应用私有缓存仓库
- 默认按账户邮箱过滤作者
- 提供失败诊断

## 阶段 5: 报告与模板系统

目标：

- 实现结构化上下文生成
- 支持默认日报模板
- 支持用户自定义模板
- 支持预览与导出

重点：

- 模板与采集解耦
- 模板错误定位
- 输出 Markdown 为第一优先格式

## 阶段 6: SSH 主机信任与连接诊断

目标：

- 实现应用私有 `known_hosts`
- 首次主机指纹确认
- 主机指纹变化告警
- 连接测试面板

重点：

- 不能默认信任所有主机
- 错误原因要对用户可理解

## 阶段 7: 调度系统

目标：

- Linux 安装 `systemd timer`
- Windows 安装 `Task Scheduler` 任务
- 在 UI 中展示实际执行时间和统计窗口

重点：

- 执行时间和统计窗口分离
- 支持重试策略
- 支持查看最近一次任务安装结果

## 阶段 8: 运行历史与问题诊断

目标：

- 保存每次执行记录
- 保存错误码和错误明细
- 提供历史报告入口

重点：

- 错误结构化
- 允许用户快速定位是 token、SSH、仓库地址还是模板问题

## 阶段 9: AI 总结增强

目标：

- 增加 AI 总结开关
- 生成“今日总结 / 风险 / 明日建议”
- 与基础规则报告共存

重点：

- AI 结果单独存储
- AI 不作为唯一事实来源

## 阶段 10: 多模板与外部输出

目标：

- 周报模板
- 管理摘要模板
- 飞书 / 邮件 / Notion 导出

## 推荐里程碑

### 里程碑 A

交付：

- 单账户
- SSH 或 Token 二选一
- 单仓库或少量仓库
- 手动生成日报

意义：

- 跑通桌面产品闭环

### 里程碑 B

交付：

- 多账户
- 多仓库
- 作者过滤
- 自动调度
- 运行历史

意义：

- 达到个人长期可用

### 里程碑 C

交付：

- known_hosts 管理
- 高级时间窗口
- 多模板
- AI 建议

意义：

- 达到产品化水平

## 验证标准

### 里程碑 A 验证标准

- 能创建账户
- 能导入 SSH 或 Token
- 能添加远程仓库
- 能测试连接成功
- 能手动生成日报
- 能预览和导出 Markdown

### 里程碑 B 验证标准

- 能为不同仓库绑定不同账户
- 能自动执行定时任务
- 能按作者过滤生成日报
- 能记录并展示失败原因

### 里程碑 C 验证标准

- 首次 SSH 连接可确认指纹
- 主机指纹变化会阻断并提示
- 支持多个时间窗口策略
- AI 总结可独立开关
