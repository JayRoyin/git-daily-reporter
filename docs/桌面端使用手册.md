# Git Daily Reporter 桌面端使用手册

## 1. 手册说明

本文档面向当前桌面端版本的实际使用流程，覆盖以下内容：

- 如何启动桌面应用
- 每个页面的用途
- 如何配置账户、凭证、仓库和 LLM 提供方
- 如何手动生成日报
- 如何使用 AI 总结
- 常见问题与注意事项

本文档只描述当前代码里已经落地的能力，不包含尚未完成的未来规划功能。

---

## 2. 软件定位

Git Daily Reporter 桌面端是一个面向开发者的本地日报工作台，用来完成以下事情：

- 管理 Git 账户
- 管理 SSH / Token 等敏感凭证
- 绑定需要统计的仓库
- 手动生成日报
- 在本地配置 LLM 提供方后，为日报生成 AI 总结

当前界面采用：

- 左侧导航
- 右侧页面切换
- 顶部语言切换

默认语言已设置为中文。

---

## 3. 启动方式

### 3.1 进入目录

在项目根目录下进入桌面端目录：

```bash
cd apps/desktop
```

### 3.2 安装依赖

首次使用需要安装前端依赖：

```bash
npm install
```

### 3.3 启动开发模式

启动桌面应用开发模式：

```bash
npm run tauri dev
```

如果你只想查看前端页面，也可以运行：

```bash
npm run dev
```

### 3.4 构建

构建前端资源：

```bash
npm run build
```

### 3.5 测试

运行当前前端测试：

```bash
npm test
```

## 5. 仪表盘

### 5.1 页面作用

仪表盘用于快速查看当前工作区的总体状态。

### 5.2 当前显示内容

仪表盘会汇总以下信息：

- 账户数量
- 凭证数量
- 仓库数量
- LLM 提供方数量
- 最近一次日报生成记录
- 建议使用流程

### 5.3 如何理解这些卡片

#### 账户数量

表示当前已经保存了多少个 Git 账户身份。

#### 凭证数量

表示当前已经保存了多少份 SSH / Token 类型的凭证。

#### 仓库数量

表示当前已经绑定了多少个仓库。

#### LLM 提供方数量

表示当前已经保存了多少个大模型提供方配置。

#### 最近一次日报生成

如果你已经生成过日报，这里会显示：

- 报告日期
- 生成时间
- 输出路径

如果还没有生成过日报，会提示你先完成仓库绑定，然后去报告中心生成第一份日报。

## 6. 账户管理

### 6.1 页面作用

账户管理用于维护 Git 身份信息。仓库和凭证都依赖账户存在。

### 6.2 当前支持填写的字段

在“账户管理”页面中，你可以填写：

- 平台
- 显示名称
- Git 用户名
- Git 邮箱
- 默认认证方式

默认认证方式当前可选：

- `SSH`
- `HTTPS + Token`

平台当前支持：

- `GitHub`
- `GitLab`
- `Gitea`
- `Gitee`
- `Custom`

### 6.3 推荐填写方式

建议按“一个实际身份一个账户”的方式配置，例如：

- 个人 GitHub 账户
- 公司 GitLab 账户
- 自建 Git 服务账户

### 6.4 自动读取与登录验证

账户页面现在提供两个辅助动作：

- `自动读取 Git 配置`
- `验证登录`

#### 自动读取 Git 配置

点击后，应用会尝试读取本机 Git 全局配置中的：

- `user.name`
- `user.email`

并自动填入表单。

#### 内置平台验证地址

对于以下平台：

- `GitHub`
- `GitLab`
- `Gitea`
- `Gitee`

应用会直接使用内置测试地址完成验证，当前页面不再展示该地址输入框。

只有当你选择：

- `Custom`

时，才需要手动填写平台地址。

#### 验证登录

验证方式取决于当前认证方式：

- `HTTPS + Token`
  使用你输入的验证 Token 向对应平台接口发起校验
- `SSH`
  记录并验证当前 SSH 身份配置是否可用于后续仓库绑定流程

### 6.5 保存账户

填写完成后，点击：

- `保存账户`

保存成功后：

- 页面会显示“账户已保存”
- 下方账户列表会展示新账户
- 后续凭证和仓库页面可直接引用该账户

### 6.6 使用建议

推荐保证以下信息准确：

- `Git 用户名` 要与真实提交身份一致
- `Git 邮箱` 要与仓库统计时希望过滤的邮箱一致

否则后续日报统计可能与预期不一致。

## 7. 凭证管理

凭证管理页面现在由 2 部分组成：

1. 凭证录入
2. LLM 提供方设置

### 7.1 凭证录入

#### 当前支持字段

在“凭证工作区”里可以填写：

- 所属账户
- 凭证类型
- 显示名称
- 用户名提示
- 来源路径
- 密钥或令牌内容

凭证类型当前可选：

- `SSH 密钥`
- `GitHub Token`

#### 推荐录入方式

##### SSH 密钥

适用于 SSH 方式访问仓库，例如：

```text
git@github.com:owner/repo.git
```

##### GitHub Token

适用于 HTTPS 方式访问仓库，例如：

```text
https://github.com/owner/repo.git
```

#### 自动扫描 SSH 密钥

页面现在支持：

- `自动扫描 SSH 密钥`

点击后，应用会尝试扫描本机常见 SSH 私钥路径，例如：

- `~/.ssh/id_ed25519`
- `~/.ssh/id_rsa`
- 其他 `~/.ssh/id_*` 私钥文件

扫描结果会显示：

- 文件名
- 来源路径
- 掩码后的密钥显示

你可以点击：

- `导入到表单`

把扫描到的 SSH 私钥直接带入当前表单。

#### 保存凭证

填写完成后，点击：

- `保存凭证`

保存成功后页面会显示：

- `凭证已保存`

页面下方的凭证列表会出现新记录。

#### 凭证显示规则

当前凭证列表默认：

- 只显示掩码
- 不直接明文展示

你可以点击：

- `显示`
- `隐藏`

来临时查看完整内容。

### 7.2 LLM 提供方设置

#### 作用

这里用于配置 AI 总结所需的大模型提供方。

#### 当前支持的预设

当前代码内置了以下预设：

- 智谱 AI
- Kimi
- 通义千问
- 文心一言
- 豆包

#### 模型选择方式

当前版本支持两种模型来源：

1. 内置模型列表
2. 从当前 API 获取模型列表

#### 内置模型列表

当你选择某个预设提供方后，应用会自动带出该提供方对应的一组常用模型，直接通过下拉框选择，不需要再手工输入。

#### 获取模型列表

页面现在支持：

- `获取模型列表`

点击后，应用会尝试从当前提供方接口拉取可用模型列表。

如果接口支持模型列表查询：

- 下拉框会刷新成真实可用模型

如果接口不支持：

- 仍然可以继续使用内置模型列表

#### 可填写字段

- 预设提供方
- 提供方名称
- Base URL
- 模型
- API 密钥

选择预设后，系统会自动带出：

- 默认 Base URL
- 模型建议值

#### 按钮说明

- `获取模型列表`
- `测试连接`
- `保存提供方`

#### 使用顺序建议

建议按这个顺序操作：

1. 选择预设提供方
2. 检查或修改 Base URL
3. 先从下拉框选择内置模型，或点击 `获取模型列表`
4. 输入 API Key
5. 点击 `测试连接`
6. 连接成功后点击 `保存提供方`

#### 保存结果

保存成功后：

- 页面会显示 `提供方已保存`
- 下方会出现“已保存 LLM 提供方”
- 你可以设置激活项
- 也可以删除旧配置

#### 关键变化

当前版本里：

- LLM 测试连接不再依赖 WebView 内部网络请求
- AI 总结请求改由桌面端后端代理执行

这样做是为了降低 Linux 下 WebKit 网络层内部错误的概率。

## 8. 仓库管理

### 8.1 页面作用

仓库管理用于把“仓库地址、账户、凭证、默认分支、作者过滤规则”绑定在一起。

这是日报生成前的核心配置页面。

### 8.2 当前支持字段

可填写内容包括：

- 账户
- 凭证
- 仓库名称
- 本地仓库路径
- 远程仓库地址
- 默认分支
- 作者过滤

### 8.3 字段解释

#### 账户

选择这个仓库对应的 Git 身份。

#### 凭证

选择用于访问该仓库的认证凭证。

#### 仓库名称

建议填写容易识别的名称，例如：

- `git-daily-reporter`
- `company-backend`

#### 本地仓库路径

用于指定本地仓库目录位置。

#### 远程仓库地址

支持两类常见格式：

SSH：

```text
git@github.com:owner/repo.git
```

HTTPS：

```text
https://github.com/owner/repo.git
```

#### 默认分支

通常填写：

- `main`
- `master`

#### 作者过滤

可填写用于过滤统计的作者值，常见是邮箱，例如：

```text
qstdc@example.com
```

### 8.4 保存仓库

填写完成后点击：

- `保存仓库`

成功后会显示：

- `仓库已保存`

下方“已绑定仓库”列表中也会显示新纪录。

### 8.5 使用建议

建议保证：

- 仓库地址可访问
- 账户与凭证匹配
- 默认分支准确
- 作者过滤与实际提交邮箱一致

否则后续日报生成可能为空，或统计结果不准确。

## 9. 报告中心

### 9.1 页面作用

报告中心用于：

- 手动生成日报
- 查看历史日报记录
- 在可用时生成 AI 总结

### 9.2 手动生成日报

当前表单包含两个输入：

- 仓库
- 报告日期

#### 操作步骤

1. 先确保仓库已经在“仓库管理”中保存
2. 进入“报告中心”
3. 在仓库下拉框中选择一个仓库
4. 输入报告日期
5. 点击 `生成日报`

#### 生成结果

如果成功，页面会显示：

- 输出路径
- 日报正文内容

同时历史记录列表会更新。

### 9.3 AI 总结

当以下条件全部满足时，`生成 AI 总结` 按钮可用：

1. 已生成日报内容
2. 已配置至少一个 LLM 提供方

#### 操作步骤

1. 先保存至少一个 LLM 提供方
2. 回到“报告中心”
3. 先点击 `生成日报`
4. 再点击 `生成 AI 总结`

#### 结果

如果调用成功，会在页面中显示 AI 生成的总结内容。

如果失败，会显示错误提示。

### 9.4 历史记录

“最近报告”区域会显示：

- 报告日期
- 输出路径

这部分用于帮助你快速回看最近生成结果。

## 11. 推荐使用顺序

如果你第一次使用这套桌面端，建议按下面顺序操作：

1. 打开应用，确认默认显示中文界面
2. 进入“账户管理”，先添加 1 个 Git 账户
3. 点击“自动读取 Git 配置”并按需调整
4. 如有需要，点击“验证登录”
5. 进入“凭证管理”，保存 1 个 SSH 密钥或 Token 凭证
6. 如需 AI 总结，再保存 1 个 LLM 提供方
7. 进入“仓库管理”，绑定账户、凭证和仓库信息
8. 进入“报告中心”，手动生成日报
9. 如需 AI 总结，再点击 `生成 AI 总结`
10. 返回“仪表盘”查看总体状态和最近输出

## 12. 常见问题

### 12.1 为什么扫描不到 SSH 密钥

原因通常是：

- 本机 `~/.ssh` 下没有常见命名的私钥文件
- 私钥文件路径不在当前扫描范围内
- 文件权限或读取失败

解决方法：

1. 手动确认 `~/.ssh/` 下是否存在私钥
2. 如未扫描到，可直接手工粘贴密钥内容
3. 或把路径填写到“来源路径”中辅助记录

### 12.2 为什么 AI 总结按钮不能点

通常有 3 种原因：

1. 还没有生成日报正文
2. 还没有配置 LLM 提供方

### 12.3 为什么报告中心里没有仓库可选

原因通常是：

- 还没有在“仓库管理”中保存任何仓库

先去仓库管理页保存仓库后，再回到报告中心。

### 12.4 为什么验证登录失败

常见原因：

- 平台选择错误
- 平台地址填写错误
- Token 无效
- 平台 API 不可达

建议先检查：

- 平台是否选对
- 如果是 GitLab/Gitea/Custom，平台地址是否准确
- Token 是否有用户信息读取权限

### 12.5 为什么界面显示英文

当前默认已经改成中文。

如果你手动点击了顶部语言切换按钮，也可以随时再切回：

- `中文`

### 12.6 Linux 下为什么可能启动失败

桌面端基于：

- Tauri
- GTK / WebKit 相关系统库

如果系统缺少相关依赖，可能导致启动失败。请参考：

- [开发环境建议.md](/home/qstdc/Royin_Project/git-daily-reporter/docs/开发环境建议.md)
- [README.md](/home/qstdc/Royin_Project/git-daily-reporter/apps/desktop/README.md)

## 13. 当前版本能力边界

当前桌面端已经比较适合做本地配置和演示，但你需要知道这几个边界：

### 已有能力

- 左侧导航工作台界面
- 账户平台选择
- Gitee 平台支持
- 本机 Git 配置自动读取
- 账户验证入口
- 凭证保存
- SSH 自动扫描
- 敏感值默认掩码显示，支持点击查看
- 仓库保存
- LLM 提供方保存
- LLM 内置模型列表
- LLM 模型列表拉取入口
- LLM 测试连接
- 手动日报生成
- AI 总结入口
- LLM 请求后端代理

### 当前仍偏早期

- 自动调度界面还没有作为独立页面落地
- 复杂模板管理还没有单独做成专页
- 更完整的运行日志、失败诊断、调度策略还会继续演进
- 当前账户验证和 SSH 验证仍属于第一版能力，后续还可以继续增强

因此建议把当前版本理解为：

- 一个已经有真实配置与生成链路的桌面工作台基础版

## 14. 日常使用建议

为了让这套工具更稳定，建议你遵循以下习惯：

- 账户按真实身份拆分，不要把个人和公司身份混在一起
- SSH 仓库优先配 SSH 密钥，HTTPS 仓库优先配 Token
- 作者过滤尽量用稳定的 Git 邮箱
- 先测试 LLM 提供方连接，再保存
- 每新增仓库后，先做一次手动日报生成验证

这样可以尽量在早期发现：

- 仓库地址错误
- 账户绑定错误
- 凭证不匹配
- API Key 不可用
- 邮箱过滤不准确
